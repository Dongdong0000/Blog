## Block 설계
`malloc`을 사용할 때에는 크기 정보를 인자로 사용한다. 하지만 `free`를 사용할 때에는 포인터만 인자로 사용한다. 왜 크기정보를 인자로 사용 안해도 될까? 답은 memory block 설계에 있다. 가장 단순한 형태부터 생각해보자. block 크기를 알아야 하면 metadata를 사용하면 된다. 근데 크기만 필요할까? block의 실제 사용여부까지 표기하는 게 적절해 보인다. (free block인지를 알아야 나중에 이 block을 할당받을 수 있기에)

```text {linenos=false}
31                     3   2   1   0
+----------------------------------+
|      size            | 0 | 0 | a |  헤더 (4바이트)
+----------------------------------+  <- malloc 이 돌려주는 주소
|                                  |
|             payload              |  사용자가 쓰는 공간
|                                  |
+----------------------------------+
|             padding              |  정렬을 맞추려고 남기는 자리
+----------------------------------+
```
`연산 속도로 인해 payload + padding은 8의 배수를 유지하게 설계한다. payload + padding의 메모리 공간은 2진수로 표현하면 하위 3개의 비트는 항상 000이다. 따라서 하위 비트에 flag를 설정한다.`

**근데 위와 같이 header만 도입했을 때에 문제가 생긴다. 왼쪽에 있는 block의 free 여부를 확인할 수가 없다.**
- 따라서 우리는 footer까지 같이 도입해야 한다. 실제로 구현할 memory block은 아래와 같다.
```text {linenos=false}
+----------------------------------+
|      size            | 0 | 0 | a |  헤더 (4바이트)
+----------------------------------+  <- malloc 이 돌려주는 주소
|                                  |
|             payload              |  사용자가 쓰는 공간
|                                  |
+----------------------------------+
|             padding              |  정렬을 맞추려고 남기는 자리
+----------------------------------+
|      size            | 0 | 0 | a |  푸터 (헤더와 같은 값)
+----------------------------------+
```

## Block 관리

블록 하나의 모양이 정해졌으니, 이제 이것들을 어떻게 훑고 다닐지 정할 차례다.

**다음 블록으로, 이전 블록으로**

블록들은 힙에 빈틈없이 이어져 있다. 그래서 현재 블록의 주소에 크기를 더하면 그대로 다음 블록의 시작이 나온다. 리스트처럼 앞에서부터 훑을 수 있지만 링크는 헤더의 크기 안에 암묵적으로 들어 있을 뿐이다. implicit free list라는 이름이 여기서 나온다.

거꾸로 가는 건 사정이 다르다. 이전 블록이 얼마나 큰지는 그 블록의 헤더에 적혀 있는데, 헤더는 이전 블록의 맨 앞에 있다. 뒤에서 앞으로 가려면 이미 그 위치를 알아야 하는 셈이라 말이 되지 않는다.

Footer를 둔 이유가 이것이다. 현재 블록 바로 앞 워드는 언제나 이전 블록의 footer이고, 거기에 이전 블록의 크기가 적혀 있다. 그 값만큼 빼면 이전 블록의 시작에 닿는다.

```text {linenos=false}
   +---+--------------+---+---+--------------+---+
   | H |     prev     | F | H |     curr     | F |
   +---+--------------+---+---+--------------+---+
                       ^^^
                        |
        현재 블록 바로 앞 워드 = 이전 블록의 푸터
```

앞뒤로 움직일 수 있게 됐으니, 이제 할당과 해제를 각각 어떻게 처리할지 보자.

**할당 요청이 오면**

힙의 처음부터 순회하면서 비어 있고 요청한 크기가 들어갈 만한 블록을 찾는다. 처음 만난 것을 바로 쓰면 first fit, 끝까지 훑어서 가장 작게 남는 것을 고르면 best fit이다. first fit은 빠르지만 앞쪽에 자잘한 조각을 남기고, best fit은 조각을 덜 남기는 대신 매번 전체를 훑는다.

빈 블록을 찾았다고 통째로 내주면 곤란하다. 400바이트짜리 자리에 100바이트를 요청받았다면 나머지 300바이트가 쓰이지도 못한 채 묶인다. 이렇게 블록 안에서 낭비되는 공간을 내부 단편화라고 한다. 그래서 남는 부분이 최소 블록 크기보다 크면 둘로 쪼개고, 뒤쪽을 새로운 빈 블록으로 만들어 헤더와 푸터를 새로 적는다.

**해제 요청이 오면**

`free(p)`는 포인터 하나만 받는다. 앞에서 본 대로 `p` 바로 앞이 헤더이므로 거기서 크기를 읽고, 할당 비트를 0으로 바꾸면 해제 자체는 끝난다.

문제는 그다음이다. 비어 있는 블록 둘이 나란히 붙어 있는데 서로 남남으로 남아 있으면, 각각 100바이트인 자리 옆에서 150바이트 요청이 들어왔을 때 거절하게 된다. 빈 공간의 총합은 충분한데 연속된 자리가 없어 쓰지 못하는 것, 이것이 외부 단편화다.

그래서 해제할 때마다 앞뒤 블록의 할당 비트를 확인해서 비어 있으면 하나로 합친다. 이것이 coalescing이다.

여기까지가 준비다. 힙이 왜 필요한지, 블록을 어떤 모양으로 잘라 쓸지, 그것들을 어떻게 훑고 합칠지까지 정했다.

## 코드 작성

본격적인 함수에 들어가기 전에, 헤더를 읽고 쓰는 도구부터 갖춘다.

```c
#define PACK(size, alloc)  ((size) | (alloc))
#define GET(p)             (*(unsigned int*)(p))
#define PUT(p, val)        (*(unsigned int*)(p) = (val))
#define GET_SIZE(p)        (GET(p) & ~0x7)
#define GET_ALLOC(p)       (GET(p) & 0x1)
```

`PACK`은 matadata 쓰는 역할이다. 크기는 항상 8의 배수라 하위 세 비트가 비어 있으니, 할당 여부를 그냥 OR로 얹으면 된다. 꺼낼 때는 반대로 `& ~0x7`로 하위 세 비트를 떼면 크기만, `& 0x1`로 맨 아래 비트만 남기면 할당 여부만 나온다.

```c
#define HDRP(bp)       ((char *)(bp) - WSIZE)
#define FTRP(bp)       ((char *)(bp) + GET_SIZE(HDRP(bp)) - DSIZE)
#define NEXT_BLKP(bp)  ((char *)(bp) + GET_SIZE(((char *)(bp) - WSIZE)))
#define PREV_BLKP(bp)  ((char *)(bp) - GET_SIZE(((char *)(bp) - DSIZE)))
```

`bp`는 블록의 시작이 아니라 페이로드의 시작이다. `malloc`이 돌려주는 그 주소다. 그래서 헤더는 한 워드 앞이고, 이게 `free(p)`가 크기를 되찾는 경로다.

푸터 위치를 알려면 블록이 얼마나 큰지 알아야 하고, 그 크기는 헤더에 있기 때문에 헤더가 아직 안 쓰인 상태에서 `FTRP`를 부르면 엉뚱한 곳을 가리키므로, 블록을 만들 때는 반드시 헤더를 먼저 쓰고 푸터를 써야 한다.

`PREV_BLKP`는 두 워드 앞을 읽는다. 그 자리가 이전 블록의 푸터다.

### mm_init

```c
int mm_init(void)
{
    char *heap = (char *)mem_sbrk(16);
    if (heap == (void *)-1) return -1;

    PUT(heap + 0,  0);            /* 정렬용 패딩 */
    PUT(heap + 4,  PACK(8, 1));   /* 프롤로그 헤더 */
    PUT(heap + 8,  PACK(8, 1));   /* 프롤로그 푸터 */
    PUT(heap + 12, PACK(0, 1));   /* 에필로그 헤더 */
    heap_listp = heap + 8;

    if (extend_heap(CHUNKSIZE / WSIZE) == NULL) return -1;
    return 0;
}
```

16바이트를 받아 네 워드를 채운다. 힙 시작 주소를 `X`라 하면 이렇게 놓인다.

```text {linenos=false}
   +--------+
   |  0000  |  X+0   패딩 워드
   +--------+
   |  8 / 1 |  X+4   프롤로그 헤더
   +--------+  <- heap_listp = X+8
   |  8 / 1 |  X+8   프롤로그 푸터
   +--------+
   |  0 / 1 |  X+12  에필로그 헤더
   +--------+
```

**패딩 워드가 왜 맨 앞인가.** 
- 페이로드가 8의 배수에 떨어지려면 헤더가 `8k+4` 자리에 있어야 한다. 패딩 없이 프롤로그 헤더를 `X+0`에 두면 첫 블록의 페이로드가 `X+12`가 되어 정렬이 깨진다. 한 워드를 버려 프롤로그 헤더를 `X+4`로 밀면 그 뒤로 오는 모든 블록의 헤더가 `8k+4`에 놓이고, 페이로드는 자동으로 8의 배수가 된다. 실제로 첫 블록의 `bp`는 `X+16`이다.

**프롤로그가 8바이트인 이유.**
- 헤더 4 + 푸터 4, 페이로드는 0바이트다. 최소 블록 크기 16보다 작지만 문제되지 않는다. 이 블록은 할당되지도 해제되지도 않고, 오직 뒤를 돌아보는 블록에게 읽히기만 하기 때문이다. 담을 데이터가 없으니 자리도 필요 없다.

**프롤로그에 푸터가 필요한 이유.** 
- 단순한 표시가 아니라 실제로 읽힌다. `coalesce`의 첫 줄을 보자.

```c
size_t prev_alloc = GET_ALLOC(FTRP(PREV_BLKP(bp)));
```

- 첫 블록에서 이걸 부르면 `PREV_BLKP`가 `bp - 8` 자리의 값을 읽는데, 그 자리가 바로 프롤로그 푸터다. 거기에 크기 8이 적혀 있어야 `bp - 8`이 계산되어 프롤로그를 가리킨다. 푸터의 **크기 값까지** 계산에 쓰이는 셈이다. 여기가 비어 있으면 쓰레기값만큼 뛰어 힙 밖으로 나간다.

**에필로그에 푸터가 없는 이유.** 
- 크기가 0이라 푸터를 둘 자리 자체가 없다. 필요도 없다. 푸터는 뒤에서 앞으로 올 때 읽히는데, 에필로그 뒤에는 아무도 없기 때문이다.

**둘 다 할당됨으로 표시한 이유.** 
- 첫 블록을 해제하면 `coalesce`가 앞을 보고, 마지막 블록을 해제하면 뒤를 본다. 프롤로그와 에필로그가 할당됨으로 보이니 합칠 대상에서 제외된다. 만약 비어 있다고 표시했다면 힙 바깥을 빈 블록으로 착각해 크기를 더하려 들고, 그 순간 헤더가 망가진다. `coalesce`에 "여기가 힙의 처음인가"를 묻는 조건문이 하나도 없는 이유가 이것이다. 예외를 코드로 처리하는 대신 데이터로 없앴다.

**에필로그의 크기가 0인 이유.** 
- `find_fit`의 순회 조건이 `GET_SIZE(HDRP(bp)) > 0`이다. 에필로그에 닿으면 크기가 0이라 반복문이 저절로 끝난다. 종료 조건까지 이 가짜 블록이 대신 해준다.

**그리고 하나 더.** 
- `X+12`에 찍은 에필로그 헤더는 첫 블록의 `bp`인 `X+16`에서 한 워드 앞이다. 즉 `HDRP(X+16)`과 같은 자리다. 바로 다음 줄에서 `extend_heap`을 부르면 이 자리가 그대로 새 블록의 헤더가 된다. `mm_init`이 찍는 에필로그는 힙의 끝 표시인 동시에, 첫 확장을 위해 미리 깔아둔 씨앗이다.

### extend_heap
