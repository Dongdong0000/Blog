## 메모리 구조

```text {linenos=false}
높은 주소 (0xFFFFFFFF)

        +------------------------+
        |         stack          |   지역 변수, 함수 호출 프레임
        |           |            |
        |           v            |   낮은 주소 방향으로 성장
        +------------------------+
        |                        |
        |        (unused)        |
        |                        |
        +------------------------+
        |           ^            |   높은 주소 방향으로 성장
        |           |            |
        |          heap          |   malloc / free 로 관리
        +------------------------+   <- brk
        |          .bss          |   초기화 안 된 전역 / static
        +------------------------+
        |         .data          |   초기화된 전역 / static
        +------------------------+
        |         .text          |   실행 코드 (읽기 전용)
        +------------------------+

낮은 주소 (0x00000000)
```
프로그램이 실행 중에 메모리가 더 필요해지면 C에서는 `malloc`, C++에서는 `new`로 힙에서 받아 온다.

> [!NOTE]
> **왜 하필 heap에서 받아오는 걸까?**
> 다른 메모리 영역은 프로그램이 실행될 때 생기는 `고정된` 공간이다.
> 따라서 프로그램이 실행 중일 때에는 heap 영역에서만 메모리를 추가로 할당받을 수 있다.

## 힙은 스택을 침범할 수 있을까

위 그림대로라면 힙과 스택은 서로를 향해 자란다. 그러면 힙을 계속 키웠을 때 언젠가 스택과 부딪히지 않을까? 직접 확인해봤다.

**실험 1. 각 영역의 주소 찍어보기**

```c
int g_init = 1;   // .data
int g_uninit;     // .bss

int main(void) {
    int   stack_var;
    void *small = malloc(100);          // 작은 요청
    void *big   = malloc(200 * 1024);   // 큰 요청

    printf(".text  %p\n", (void*)main);
    printf(".data  %p\n", (void*)&g_init);
    printf(".bss   %p\n", (void*)&g_uninit);
    printf("brk    %p\n", sbrk(0));
    printf("small  %p\n", small);
    printf("big    %p\n", big);
    printf("stack  %p\n", (void*)&stack_var);
}
```

```text {linenos=false}
.text  0x5a7a873c41a9
.data  0x5a7a873c7010
.bss   0x5a7a873c7018
brk    0x5a7aa5285000
small  0x5a7aa52642a0
big    0x73e3c302b010     <- 힙이 아니다
stack  0x7ffd13ca8144

stack - brk = 38409.7 GB
```

`.text`, `.data`, `.bss`, 힙은 그림대로 낮은 주소에 차례로 붙어 있다. 그런데 스택까지의 거리가 38TB다. 그림에서 얇은 띠처럼 그린 `(unused)` 구간이 실제로는 나머지 전부를 합친 것보다 압도적으로 넓다.

`big`도 눈에 띈다. 200KB를 요청했더니 힙이 아니라 완전히 다른 주소에 잡혔다. glibc는 일정 크기(기본 128KB) 이상의 요청을 `brk`가 아니라 `mmap`으로 처리한다. 힙이 위로 쭉 자란다는 그림은 작은 할당에만 해당하는 이야기다.

**실험 2. brk를 끝까지 밀어보기**

`sbrk`로 1GB씩 힙을 늘려서 실패할 때까지 반복했다.

```c
long step = 1L << 30;
long total = 0;
while (sbrk(step) != (void*)-1) total += step;
```

```text {linenos=false}
최종 brk  : 0x7867cf1ad000
늘린 양   : 29373 GB
스택 주소 : 0x7fffeacca0fc
남은 거리 : 7776.4 GB

malloc(1TB) = (nil)  -> NULL 반환
```

29TB를 늘리고 멈췄는데, 스택까지는 아직 7.7TB가 남아 있었다. 스택 근처에도 못 갔다는 뜻이다. 그러면 무엇에 막힌 걸까? 멈춘 지점에서 `/proc/self/maps`를 열어봤다.

```text {linenos=false}
5c4c57e33000-78ff97e54000 rw-p ...  [heap]
78ff98000000-78ff98028000 r--p ...  /usr/lib/x86_64-linux-gnu/libc.so.6
78ff98028000-78ff981bd000 r-xp ...  /usr/lib/x86_64-linux-gnu/libc.so.6
```

힙 끝이 `0x78ff97e54000`, 바로 다음에 libc가 `0x78ff98000000`부터 매핑되어 있다. 둘 사이 간격은 약 1.7MB. 힙은 스택이 아니라 **공유 라이브러리가 올라간 mmap 영역**에 막혀 멈춘 것이다.

위 그림은 각 영역의 역할과 성장 방향을 이해하는 용도로는 유효하지만, 실제 주소 공간의 축척과는 다르다는 점을 기억해두면 된다.

```
응용 프로그램          p = malloc(100)
                              │
  ────────────────────────────┼──────────────────────  user space
                              ▼
  libc (ptmalloc)      free list에 재사용할 블록 있나?
                              │
                    ┌─────────┴─────────┐
                   있다                 없다       
              주소 반환             128KB 미만 → brk()
              (여기서 끝,           128KB 이상 → mmap()
               커널 안 감)
                                          │
  ════════════════════════════════════════│════════  트랩 (시스템 콜)
                                          ▼
  커널
```
`sbrk 같은 시스템 콜은 커널이 제공하는 것을 그대로 쓰고, 그렇게 받아온 덩어리를 어떻게 쪼개고 합칠지를 구현한다. 시스템 콜은 비싸다. 그래서 요청이 올 때마다 커널에 들어가는 대신, 한 번에 넉넉히 받아두고 그 안에서 잘라 쓴다.`


## 그래서 무엇을 만들어야 하나
이제 sbrk로 큰 덩어리를 받아왔다고 하자. 여기서부터는 우리 몫이다.
100바이트를 요청받으면 이 덩어리의 어디를 잘라 줄지 정해야 하고, 나중에 free(p)로 돌아오면 그 자리를 다시 쓸 수 있게 표시해둬야 한다. 그런데 free는 포인터 하나만 받는다. 크기는 알려주지 않는다. 얼마를 반납받은 건지 어떻게 알 수 있을까?
